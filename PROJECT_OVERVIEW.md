# garden_project — Project Overview

A plain-language explanation of every component in the stack: what it is, why it was chosen, how it was set up, and how it fits into the whole.

---

## The big picture

The project is a full-stack web application that tracks and visualises a real garden. It has three visible pages:

| URL | Who sees it | What it does |
|---|---|---|
| `/` | Anyone | Public read-only map with hover tooltips |
| `/gardener` | You | Interactive map + plant cards + quick-log form + journal |
| `/dashboard` | You | Health scores, stats, plant list sorted worst-first |

Behind those pages is a data pipeline that runs every day to record snapshots of plant health and generate maintenance tasks automatically.

---

## How the pieces connect

```
Browser
  └── Caddy (reverse proxy, HTTPS)
        └── FastAPI (Python web server, port 8000)
              ├── Serves HTML / CSS / JS files
              ├── Reads/writes PostgreSQL (live data)
              └── Reads dbt_marts schema (computed scores)

PostgreSQL (database, port 5432)
  ├── public schema  ← live data written by FastAPI
  ├── dbt_staging    ← views created by dbt (read-only)
  └── dbt_marts      ← tables created by dbt (computed scores)

dbt  ← runs locally, transforms public → dbt_marts
Dagster ← runs locally, daily pipeline: snapshot + generate tasks
GitHub Actions ← deploys to VPS on every push to main
```

Everything except dbt and Dagster runs inside Docker containers on both your local machine and the VPS.

## Component 1 — PostgreSQL (the database)

**What it is:** A relational database. Think of it as a collection of spreadsheet-like tables where rows are records and columns are fields. All the real data lives here.

**Why PostgreSQL:** It's the industry standard for structured data, handles relationships between tables (e.g. a plant belongs to a zone), and has excellent support in every tool we use.

**How it was set up:** It runs as a Docker container defined in `docker-compose.yml`. The database name, user, and password come from your `.env` file. The container stores data in a Docker volume called `postgres_data` so data survives restarts.

**The tables (public schema):**

| Table | What it stores |
|---|---|
| `zones` | Areas of the garden (lawn, patio, pond, etc.) with conditions (light, moisture, pH) |
| `plants` | Every plant: name, latin name, status, position on the map, container type |
| `journal_entries` | Your log entries (actions, observations, purchases, decisions) |
| `tasks` | To-do items, both manual and auto-generated by Dagster |
| `plant_status_history` | Daily snapshots of each plant's status — the historical record |
| `gardener_profile` | Your XP and level (RPG mechanic, not yet fully wired) |
| `gardener_levels` | XP thresholds and titles for each level |
| `garden_levels` | Vitality score thresholds for garden levels |
| `achievements` | Badges you can unlock |
| `gardener_achievements` | Which badges you have earned |

**The data in it:** 51 plants were seeded from `backend/seeds/seed_plants.py`. Zones were seeded from `backend/seeds/seed_reference_data.py`. Seeding is a one-time operation — it inserts the initial real data.

## Component 2 — Alembic (database migrations)

**What it is:** A tool that manages changes to the database structure over time. Instead of manually running `ALTER TABLE` commands, you write migration files that Alembic applies in order.

**Why it matters:** If you add a new column to a model in Python, the database doesn't automatically know about it. Alembic bridges that gap.

**How it works:**
1. You change a SQLAlchemy model in `backend/app/models/`
2. You run `alembic revision -m "description"` — it generates a migration file in `backend/migrations/versions/`
3. You run `alembic upgrade head` — it applies all pending migrations to the database

**Key files:**
- `backend/alembic.ini` — tells Alembic where the database is (uses `localhost:5432` because Alembic runs on your machine, not inside Docker)
- `backend/migrations/versions/` — one file per migration, each with an `upgrade()` and `downgrade()` function

**Commands (always run locally, inside the venv):**
```bash
cd backend
source .venv/Scripts/activate
alembic upgrade head        # apply pending migrations
alembic revision -m "desc"  # create a new migration
alembic downgrade -1        # undo the last migration
```

## Component 3 — FastAPI (the backend / web server)

**What it is:** A Python web framework. It handles two jobs: serving the HTML pages to the browser, and providing a JSON API that the JavaScript on those pages calls to read/write data.

**Why FastAPI:** Fast to write, automatically generates interactive API documentation at `/docs`, and handles async requests well.

**How it's structured:**

```
backend/app/
  main.py          ← entry point: registers all routers, serves HTML files
  database.py      ← database connection (SQLAlchemy session)
  models/          ← Python classes that describe the database tables
  routers/         ← one file per resource (plants, zones, journal, tasks, dashboard)
```

**What a router does:** Each router file defines URL endpoints. For example, `routers/plants.py` defines:
- `GET /api/plants/` → returns all active plants as JSON
- `GET /api/plants/{id}` → returns one plant
- `POST /api/plants/` → creates a plant
- `PATCH /api/plants/{id}` → updates a plant

The browser's JavaScript calls these endpoints to load data and submit forms.

**SQLAlchemy models:** The files in `models/` are Python classes where each class = one database table and each attribute = one column. SQLAlchemy translates between Python objects and SQL queries so you rarely write raw SQL in the application code.

**Hot reload:** FastAPI restarts automatically when you save a Python file. No need to restart Docker for Python changes. Static files (HTML/CSS/JS) are served directly and update immediately on browser refresh.

**How to access the API docs:** Go to `http://localhost:8000/docs` while the stack is running. You can browse and test every endpoint interactively.

## Component 4 — Docker Compose (running the services)

**What it is:** A tool that runs multiple services (database, web server, proxy) together as a group, with the right configuration and connections between them.

**Why Docker:** Guarantees that the app runs identically on your laptop and on the VPS, regardless of what software is installed on the host machine. Each service runs in its own isolated container.

**The three services in `docker-compose.yml`:**

| Service | Image | What it does |
|---|---|---|
| `db` | `postgres:16` | The PostgreSQL database |
| `backend` | Built from `backend/Dockerfile` | The FastAPI Python server |
| `caddy` | `caddy:2` | The reverse proxy (see below) |

**How they connect:** The backend connects to the database using the hostname `db` (Docker's internal DNS) on port 5432. From outside Docker (your browser, Alembic, dbt), you connect on `localhost:5432`.

**Volumes:** Persistent storage that survives container restarts:
- `postgres_data` — database files
- `media_data` — uploaded images
- `caddy_data` / `caddy_config` — Caddy's certificates and config

**The `.env` file:** Contains secrets that Docker injects as environment variables:
```
DB_PASSWORD=...      ← PostgreSQL password
SECRET_KEY=...       ← used for future auth (JWT signing)
ENVIRONMENT=...      ← development or production
```
This file is never committed to git.

**Common commands:**
```bash
docker compose up -d           # start everything in background
docker compose down            # stop everything
docker compose up -d --build   # rebuild after Dockerfile changes
docker compose logs -f         # stream logs from all services
```

## Component 5 — Caddy (the reverse proxy)

**What it is:** A web server that sits in front of FastAPI and handles HTTPS automatically.

**Why it's needed:** Browsers require HTTPS for many features and for trust. FastAPI itself doesn't handle HTTPS certificates — that's Caddy's job. Caddy obtains and renews free certificates from Let's Encrypt automatically on the VPS.

**What it does:**
- On the VPS: listens on ports 80 and 443, obtains an SSL certificate for `garden.lauramarion.eu`, and forwards all requests to FastAPI on port 8000
- Locally: passes requests through to FastAPI without HTTPS (using `Caddyfile.dev`)

**You almost never need to touch it** unless you're adding a new domain or changing ports.

## Component 6 — The frontend (HTML + CSS + JS)

**What it is:** Three plain HTML pages with one shared CSS file and separate JavaScript files. No framework (no React, no Vue) — just standard web technologies.

**Why no framework:** The app is simple enough that a framework would add complexity without benefit. Plain JS is easier to understand and has no build step.

**File structure:**
```
backend/static/
  css/
    main.css         ← ALL styles for the entire app (one file, no exceptions)
  js/
    topbar.js        ← shared topbar component (injected into every page)
    map.js           ← isometric map: drawing, hit detection, tooltips
    gardener.js      ← gardener page: plant cards, log form, journal feed
    dashboard.js     ← dashboard page: vitality score, plant HP list
  public.html        ← public read-only page
  gardener.html      ← main gardener interface
  dashboard.html     ← health dashboard
```

**How pages load data:** Each JS file calls `fetch('/api/...')` to get data from FastAPI as JSON, then builds HTML strings and inserts them into the page. There is no page reload — it's all dynamic after the initial HTML loads.

**The shared topbar:** `topbar.js` injects the full header (title + nav + HUD) into every page. It reads `window.location.pathname` to know which nav link to mark as active. Future pages just need `<header id="topbar"></header>` + a script tag.

## Component 7 — The isometric map (canvas)

**What it is:** A grid drawn on an HTML `<canvas>` element using JavaScript. No mapping library — it's custom-drawn geometry.

**How the grid works:** The garden is a 19×10 grid of tiles (columns −1 to 17, rows 0 to 9). Each tile is a diamond shape drawn with 4 lines. The isometric effect comes from offsetting X and Y by the tile's column and row position using the `toScreen(c, r)` function.

**The coordinate system:**
- **Letters A–J** = the row axis (A = row 0 = front of garden, J = row 9 = back)
- **Numbers 1–19** = the column axis (1 = col 17 = patio front, 19 = col −1 = ivy fence at the back)
- Example: tile **E4** = column 14, row 4 (in the patio uncovered zone)
- This is the system to use when describing plant positions — never raw col/row numbers

**Zone colours:** Each tile is coloured based on `zoneCodeAt(c, r)` — a function that returns the zone name for any tile coordinate. The ivy fence (col −1), pond, stairs, patio shelves etc. all have specific colours defined in `ZONE_COLORS`.

**Plants on the map:** Plants are fetched from `/api/plants/` and drawn as coloured circles with HP bars above them. Up to 5 plants can share a tile using `grid_slot` (0–4), each offset slightly using `SLOT_OFFSETS`.

**Hit detection:** When you hover the mouse, `plantAtMouse()` finds the nearest plant within 22px and highlights it. Clicking calls `onPlantClick()` which gardener.js overrides to open the plant's detail card.

## Component 8 — The design system (`design_system.md`)

A locked set of visual rules that every page follows:

| Rule | Details |
|---|---|
| **Fonts** | Silkscreen (bold, uppercase) for all labels/headings; Inter for body text |
| **Colours** | Seafoam page background → White cards → Mint inset → Lavender badges |
| **Colour = meaning** | Green = healthy, Yellow = warning, Pink = danger/lost — never decorative |
| **No shadows** | Depth comes from background layering only |
| **No emoji** | Pixel art icons only (from PixelForge tool) |
| **Spacing** | Multiples of 4px only |

The CSS variables in `:root` in `main.css` are the single source of truth for all colours, radii, and fonts.

## Component 9 — dbt (data transformation)

**What it is:** A tool that runs SQL queries to transform your raw data into cleaner, more useful tables. It's the "T" in ETL (Extract, Transform, Load).

**Why it's needed:** The raw `plants` table stores `status` as a text field (OK/WARNING). The HP score used on the dashboard and the map is a *computed* value, not stored directly. dbt computes these scores and writes them into separate tables that the app can query.

**How it works:**
1. You write `.sql` files in `dbt/models/` — each file is one table or view
2. You run `dbt run` and it executes those SQL files against the database
3. The results appear as new tables/views in separate schemas

**The models:**

```
dbt/models/
  staging/           ← views that clean/rename raw tables
    stg_plants.sql          → renames columns, filters is_active=true
    stg_journal_entries.sql → renames entry id to entry_id
    stg_tasks.sql           → renames source to task_source (avoids SQL keyword)
    sources.yml             → tells dbt which raw tables exist

  marts/             ← final tables the app queries
    plant_hp.sql            → computes HP score per plant (currently: OK=80, WARNING=35, else=5)
    garden_vitality_score.sql → computes overall garden score (average HP across all plants)
```

**The schemas created in PostgreSQL:**
- `dbt_staging` — staging views (fast, no data stored, recalculated on every query)
- `dbt_marts` — mart tables (data is physically stored, fast to query)

**How to run:**
```bash
cd dbt
dbt run --profiles-dir .    # rebuild all models
dbt run --profiles-dir . -s plant_hp   # rebuild just one model
```

**The `profiles.yml` file** (in `dbt/`, never committed) contains the database connection details. It uses the same PostgreSQL credentials as the app.

**When to re-run dbt:** Whenever plant statuses change significantly, or on a schedule. Dagster (below) is meant to automate this eventually.

## Component 10 — Dagster (the daily pipeline)

**What it is:** An orchestration tool — it runs tasks (called "assets") in a defined order on a schedule.

**Why it's needed:** Some work shouldn't happen on every web request — it should happen once a day in the background. Specifically: recording daily snapshots of plant health, and generating maintenance tasks automatically.

**How it works:** Dagster has a web UI at `http://localhost:3000`. From there you can see the pipeline, trigger runs manually, and enable the daily schedule.

**The two assets:**

1. **`plant_status_snapshot`**
   Writes one row to `plant_status_history` for every active plant, recording their current status. Skips plants already snapshotted today (so it's safe to run multiple times). This builds the historical record that will eventually power richer HP calculations.

2. **`generate_tasks`** *(depends on snapshot completing first)*
   Applies two rules to auto-generate tasks:
   - **Rule 1:** Plant has `WARNING` status and no pending auto task → creates "Investigate: {name}" (urgent)
   - **Rule 2:** Plant has had no journal action in 14 days and no pending check task → creates "Check on: {name}" (soon)
   Both rules are idempotent — they won't create duplicate tasks.

**The schedule:** Runs every day at 8am (`0 8 * * *` in cron syntax).

**How to start Dagster:**
```bash
cd dagster
dagster dev -w workspace.yaml
```

**Key files:**
```
dagster/
  workspace.yaml                  ← tells Dagster where to find the code
  garden_pipeline/
    __init__.py                   ← the Definitions object (registers everything)
    resources.py                  ← database connection (reads .env)
    assets/
      snapshot.py                 ← plant_status_snapshot asset
      tasks.py                    ← generate_tasks asset
    schedules.py                  ← daily_schedule definition
```

## Component 11 — GitHub Actions (CI/CD)

**What CI/CD means:** Continuous Integration / Continuous Deployment. Every time you push code to the `main` branch on GitHub, a script runs automatically to deploy it to the VPS. You never need to SSH in just to deploy.

**How it works:**
1. You run `git push` from your local machine
2. GitHub detects the push to `main` and triggers the workflow defined in `.github/workflows/deploy.yml`
3. The workflow SSHes into the VPS using a private key stored as a GitHub Secret
4. On the VPS it runs: `git pull` → `docker compose up -d --build` → `alembic upgrade head`

**The three GitHub Secrets** (stored in your repo's Settings → Secrets):
- `VPS_HOST` — the VPS IP address
- `VPS_USER` — `ubuntu`
- `VPS_SSH_KEY` — your private SSH key (the VPS's `authorized_keys` has the matching public key)

**What happens on the VPS:** The VPS pulls the latest code from GitHub and rebuilds the Docker containers. It never originates code — it only receives what GitHub sends.

## Component 12 — The local development workflow

**Golden rule:** All code is written and edited locally. The VPS only receives changes via GitHub.

**Starting the local stack:**
```bash
docker compose up -d        # start db + backend + caddy
```
After this, the app is at `http://localhost:8000`.

**Python tools** (Alembic, dbt, seeds) run locally using the virtual environment:
```bash
cd backend
source .venv/Scripts/activate   # activate the venv (Windows Git Bash)
```

**Changing Python code:** FastAPI hot-reloads automatically. Save a `.py` file and the change is live immediately.

**Changing HTML/CSS/JS:** Save the file and hard-refresh the browser (`Ctrl+Shift+R`). No restart needed.

**Adding a new database column:**
1. Edit the model in `backend/app/models/`
2. Run `alembic revision -m "description"` to generate a migration
3. Run `alembic upgrade head` to apply it

**Deploying:**
```bash
git add .
git commit -m "feat: description"
git push    # ← this triggers the GitHub Action and deploys to the VPS
```

## Data flow summary

Here is how data moves through the full system on a typical day:

```
You log an action in the browser
  → JavaScript POSTs to /api/journal/
  → FastAPI writes a row to journal_entries in PostgreSQL

At 8am, Dagster wakes up and runs the daily pipeline:
  → plant_status_snapshot writes today's status for all 51 plants to plant_status_history
  → generate_tasks scans for WARNING plants and neglected plants, creates tasks in the tasks table

You run dbt run:
  → dbt reads plants + journal_entries from public schema
  → Computes HP per plant and overall vitality score
  → Writes results to dbt_marts.plant_hp and dbt_marts.garden_vitality_score

You open the dashboard:
  → JavaScript fetches /api/dashboard/summary → reads from dbt_marts.garden_vitality_score
  → JavaScript fetches /api/dashboard/plants → reads from dbt_marts.plant_hp
  → Page renders with current scores
```
