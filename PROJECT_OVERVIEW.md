# garden_project — Project Overview

A plain-language explanation of every component in the stack: what it is, why it was chosen, how it was set up, and how it fits into the whole.

---

## The big picture

The project is a full-stack web application that tracks and visualises a real garden.

| URL | What it does |
|---|---|
| `/` | Public read-only map with hover tooltips |
| `/gardener` | Interactive map + side panel (plant list, zone info, quick-log form) |
| `/dashboard` | Vitality score, HP stats, XP bar, disturbances, recent journal |
| `/plants` | All plants in a filterable grid with Add plant modal |
| `/plants/{id}` | Single plant detail — info, journal, pending tasks |
| `/journal` | All journal entries grouped by season, filterable |
| `/tasks` | Task list bucketed by urgency (overdue / this week / upcoming) |

Behind those pages is a data pipeline that runs every day to record snapshots of plant health and generate maintenance tasks automatically.

---

## How the pieces connect

```
Browser
  └── Caddy (reverse proxy, HTTPS)
        └── FastAPI (Python web server, port 8000)
              ├── Serves HTML / CSS / JS files
              ├── Reads/writes PostgreSQL (live data)
              └── Reads dbt_marts schema (computed scores)

PostgreSQL (database, port 5432)
  ├── public schema  ← live data written by FastAPI
  ├── dbt_staging    ← views created by dbt (read-only)
  └── dbt_marts      ← tables created by dbt (computed scores)

dbt       ← runs locally, transforms public → dbt_marts
Dagster   ← runs locally, daily pipeline: snapshot + generate tasks
GitHub Actions ← deploys to VPS on every push to main
```

Everything except dbt and Dagster runs inside Docker containers on both your local machine and the VPS.

## Component 1 — PostgreSQL (the database)

**What it is:** A relational database. All the real data lives here.

**The tables (public schema):**

| Table | What it stores |
|---|---|
| `zones` | Areas of the garden (lawn, patio, pond, etc.) with conditions (light, moisture, pH) |
| `plants` | Every plant: name, latin name, vitality status, position on the map, container type |
| `journal_entries` | Your log entries (actions, observations, purchases, decisions) |
| `tasks` | To-do items, both manual and auto-generated by Dagster |
| `plant_status_history` | Daily snapshots of each plant's vitality status |
| `gardener_profile` | Your XP and level (RPG mechanic, partially wired) |
| `gardener_levels` | XP thresholds and titles for each level |
| `garden_levels` | Vitality score thresholds for garden levels |
| `achievements` | Badges you can unlock |
| `gardener_achievements` | Which badges you have earned |

**The data:** 51 plants seeded from `backend/seeds/seed_plants.py`. Zones from `backend/seeds/seed_reference_data.py`. Seeding is a one-time operation.

## Component 2 — Alembic (database migrations)

**What it is:** A tool that manages changes to the database structure over time. Whenever you add a column or table, you write a migration file and Alembic applies it.

**Commands (always run locally, inside the venv):**
```bash
cd backend
source .venv/Scripts/activate
alembic upgrade head        # apply pending migrations
alembic revision -m "desc"  # create a new migration
alembic downgrade -1        # undo the last migration
```

Uses `localhost:5432` (not `db`) because Alembic runs on your machine, not inside Docker.

**Migrations so far:**
1. `36c0bf018efb` — initial schema (all 10 tables)
2. `fd8481ae110a` — add `has_soil` and `soil_ph` to zones
3. `a1b2c3d4e5f6` — rename vitality states (OK→Thriving, WARNING→Struggling) + shift grid_slot to 1-indexed

## Component 3 — FastAPI (the backend / web server)

**What it is:** A Python web framework that serves HTML pages and provides a JSON API.

**Structure:**
```
backend/app/
  main.py          ← entry point: registers all routers, serves HTML pages
  database.py      ← database connection (SQLAlchemy session)
  models/          ← Python classes = database tables
  routers/
    plants.py      ← GET/POST/PATCH /api/plants/
    zones.py       ← GET /api/zones/
    journal.py     ← GET/POST /api/journal/ (filters: plant_id, zone_id, entry_type)
    tasks.py       ← GET/POST/PATCH /api/tasks/ (filters: plant_id, zone_id, status)
    dashboard.py   ← GET /api/dashboard/summary + /api/dashboard/plants
    gardener.py    ← GET /api/gardener/profile (XP + level)
```

**API docs:** `http://localhost:8000/docs` — browse and test every endpoint interactively.

## Component 6 — The frontend (HTML + CSS + JS)

**Plain HTML + CSS + Vanilla JS — no framework, no build step.**

**File structure:**
```
backend/static/
  css/main.css          ← ALL styles for the entire app (one file, no exceptions)
  icons/                ← 15 pixel art SVG icons
  sprites/              ← plant sprites (64×64 PNG, Stardew Valley style)
  js/
    topbar.js           ← shared header injected into every page
    map.js              ← isometric canvas drawing, hit detection, tooltips
    gardener.js         ← map side panel: plant list, zones, log form
    dashboard.js        ← vitality score, HP list, XP bar, recent journal
    tasks.js            ← task buckets, add form, mark done
    journal.js          ← season-grouped entries, filters, new entry form
    plants.js           ← plant grid, filters, add plant modal (3-step wizard)
    plant_detail.js     ← single plant view, mark as lost
  public.html           ← public read-only map
  gardener.html         ← map + Plants / Zones / Log side panel
  dashboard.html        ← health dashboard
  tasks.html            ← task management
  journal.html          ← journal feed
  plants.html           ← plant grid + add modal
  plant_detail.html     ← single plant detail page
```

**How pages load data:** Each JS file calls `fetch('/api/...')`, builds HTML strings, and inserts them into the page. No page reloads.

**The shared topbar:** Every page has `<header id="topbar"></header>` + a `<script src="/static/js/topbar.js">` tag. The script injects the full header: title + 5-item nav (Map, Plants, Journal, Tasks, Dashboard) + season badge + HUD. On mobile (<768px) the desktop nav hides and a bottom tab bar appears.

**Vitality states:** Six states with associated HP values used throughout the UI:
| State | HP | Colour meaning |
|---|---|---|
| Thriving | 80 | Green |
| Stable | 65 | Lavender |
| New | 60 | Aqua |
| Dormant | 50 | Lavender |
| Struggling | 35 | Yellow |
| Lost | 5 | Pink |

## Component 7 — The isometric map (canvas)

**Custom-drawn geometry on an HTML `<canvas>` — no library.**

**The grid:** 19×10 tiles (columns −1 to 17, rows 0 to 9). Column −1 is the ivy fence (dark green, back of garden). Each tile is a diamond drawn with 4 lines using `toScreen(col, row)`.

**Coordinate system:**
- **Letters A–J** = row axis (A = row 0 = front, J = row 9 = back)
- **Numbers 1–19** = column axis (1 = col 17 = patio front, 19 = col −1 = ivy fence)
- Example: **E4** = col 14, row 4. Always use this format — never raw numbers.

**GRID_SLOTS:** Up to 5 plants share a tile using 1-indexed slots — 1=centre, 2=back, 3=front, 4=left, 5=right.

**Map side panel (gardener page):** Three tabs:
- **Plants** — scrollable list of all plants; click any row to highlight on the map and show detail card
- **Zones** — list of all zones with light/moisture conditions
- **Log** — quick journal entry form

## Component 8 — The design system (`design_system.md`)

Locked visual rules all pages follow:

| Rule | Details |
|---|---|
| **Fonts** | Silkscreen (labels/headings) + Inter (body text) |
| **Colours** | Seafoam page bg → White cards → Mint inset → Lavender badges |
| **Colour = meaning** | Green = healthy, Yellow = warning, Pink = danger — never decorative |
| **No shadows** | Depth from background layering only |
| **No emoji** | SVG pixel art icons only |
| **Spacing** | `--space-xs` through `--space-3xl` (4px base) |

All tokens live in `:root` in `main.css`.

## Component 9 — dbt (data transformation)

**Computes HP scores and garden vitality — values never stored raw in the plants table.**

**Models:**
```
dbt/models/
  staging/
    stg_plants.sql             → filters active plants, renames id→plant_id
    stg_journal_entries.sql    → renames id→entry_id
    stg_tasks.sql              → renames source→task_source
    sources.yml                → declares raw tables

  marts/
    plant_hp.sql               → HP by vitality status + last action date
    garden_vitality_score.sql  → avg HP, plant counts by state
```

**Schemas created in PostgreSQL:** `dbt_staging` (views) and `dbt_marts` (tables).

**Run:**
```bash
cd dbt
dbt run --profiles-dir .
```

## Component 10 — Dagster (the daily pipeline)

**Runs two assets every day at 8am.**

1. **`plant_status_snapshot`** — writes one row to `plant_status_history` per active plant (today's vitality status). Idempotent.

2. **`generate_tasks`** — auto-generates maintenance tasks:
   - Plant is `Struggling` + no pending auto task → "Investigate: {name}" (urgent)
   - No journal action in 14 days + no pending check task → "Check on: {name}" (soon)

**Start locally:**
```bash
cd dagster
dagster dev -w workspace.yaml   # UI at http://localhost:3000
```

## Component 12 — Local development workflow

```bash
docker compose up -d        # start db + backend + caddy → app at localhost:8000
```

**Python changes:** FastAPI hot-reloads automatically.  
**HTML/CSS/JS changes:** Hard-refresh the browser (`Ctrl+Shift+R`).  
**New DB column:** Edit model → `alembic revision -m "desc"` → `alembic upgrade head`.  
**Deploy:** `git push` — the GitHub Action handles everything.

## Data flow summary

```
You log an action in the browser
  → JS POSTs to /api/journal/
  → FastAPI writes to journal_entries in PostgreSQL

At 8am, Dagster runs:
  → plant_status_snapshot → writes to plant_status_history
  → generate_tasks → scans for Struggling plants + neglected plants → creates tasks

You run dbt:
  → Reads plants + journal_entries
  → Computes HP per plant + garden vitality score
  → Writes to dbt_marts.plant_hp and dbt_marts.garden_vitality_score

You open the dashboard:
  → Fetches /api/dashboard/summary → reads dbt_marts.garden_vitality_score
  → Fetches /api/dashboard/plants → reads dbt_marts.plant_hp
  → Fetches /api/gardener/profile → reads gardener_profile
  → Page renders with current scores, disturbances, and recent journal
```
